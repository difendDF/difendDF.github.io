# Git Stash

真是无语住了，之前还写了 git 的操作，然后这个命令竟然完全没看...

用到这个命令的场景大概可以是这样：在一个分支上做了一些修改，这时候想要切换到另一个分支上去做一些事情，但是当前分支上的工作并没有做完，也不想为这次切换创建一次新的提交，就可以使用 git stash 命令来暂存我们现在的修改。

# 注意事项

git stash 的数据模型是基于栈的，但是他到底是基于整个 git 工作空间的栈，还是当前分支的保存栈呢？答：存储栈是基于整个工作空间的，同时储存文件操作是当前操作分支维度的。
什么意思？比如我现在平台内有两个分支 branch1 和 branch2，然后我在 branch1 上执行了 git stash 命令，这时候我切换到 branch2 上执行 git stash pop ，然后我之前在 branch1 上的修改就到了 branch2 上了。

# 项目封装与迭代

其实我这个周的实习还有收获比较多的就是对于项目中组件的抽象与封装有了一定的认识与思考。之前后端开发的时候虽然有做过一些封装，但其实抽象出来公共父类然后继承不同的子类的场景和经验还是比较少的。
在现在的这个项目中，很奇妙的就是一个页面会被分成好几个部分 ——Container、page、pageComponent 等等。我当时看代码的时候能够理解把页面中的一些组件抽象出来封装成单独的 component，但是对于还要弄一个 container 的操作表示十分的不理解。现在想想发现这种操作还是挺好的，我在做页面路由跳转的时候就可以使用同一个 Container 来写，然后通过传入的参数值不同，来决定 container 中显示哪些 page 的内容，仔细想想感觉确实是使项目的耦合度更低了。
然后我就在佩服说，我靠，人家好牛啊，竟然能想到这种抽象封装的方式。直到我去看了 pr 的提交记录，发现其实一开始的代码也是都写在一起的，不过在后期 feature 增多的情况下，发现有些公共的 component 是可以复用的，然后才开始抽象出新的 component，然后就成就了现在项目的结构。所以在这个过程中，我对整个项目的迭代也是有了一定的体会，希望自己以后写代码不要总想着一步到位，别逼狠了，迭代的过程也是蛮有趣的。
